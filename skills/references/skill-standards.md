# Skill Standards — Anthropic-Compliant SKILL.md

Standards derived from [Anthropic's official skill-creator](https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md)
and [Claude prompting best practices](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/claude-prompting-best-practices).

Every skill generated by a creator must comply with these standards.

## Skill anatomy

```
skill-name/
├── SKILL.md          ← Required — instructions for the agent
├── references/       ← Optional — deep-dive docs loaded on demand
├── scripts/          ← Optional — executable code for deterministic tasks
└── assets/           ← Optional — templates, icons, fonts
```

## YAML frontmatter (required)

```yaml
---
name: skill-name
description: >
  What it does. Use when [triggers]. Activates for [keywords].
  Include broad trigger phrases so the skill activates reliably.
---
```

**Description rules:**
- Include WHAT the skill does AND WHEN to use it
- Be "pushy" — Claude tends to under-trigger skills, so over-specify activation contexts
- Include specific trigger words and phrases users might say
- All "when to use" info belongs here, not only in the body

## Body standards

### Length

- **Target: under 500 lines.** If approaching this limit, move detail into `references/`
  with clear pointers about when to read them.
- Progressive disclosure: metadata (~100 words, always in context) → body (loaded on trigger)
  → references (loaded on demand, unlimited size)

### Writing style

1. **Imperative form.** "Trace the execution path" not "You should trace the execution path."

2. **Explain WHY, not just WHAT.** "Check for null because JavaScript's `!!0` returns false,
   hiding valid zero values" teaches the model to reason about edge cases.

3. **Keep it lean.** Remove instructions that don't change behavior. If the model would do
   something anyway, the instruction is noise.

4. **Be general.** Describe patterns, not instances. Avoid narrowing to specific examples
   that become brittle.

5. **Reason over rules.** Instead of "ALWAYS do X", explain why X matters. Models with good
   theory of mind produce better output from reasoning than from rigid rules.

6. **Phases over checklists.** Group related steps into named phases rather than flat numbered
   lists. Models handle hierarchical structure better.

### Structure

Every SKILL.md should have:

| Section | Purpose | Required? |
|---------|---------|-----------|
| **Title + summary** | `# Name` with one-line description | Yes |
| **When to use** | Table or bullets: use vs. not-use | Yes |
| **Core workflow** | Main content, organized in phases | Yes |
| **Examples** | `<example>` blocks showing realistic usage | Yes |
| **Tips** | Practical advice, gotchas | Recommended |
| **Anti-patterns** | What to avoid and why | Recommended |

### Examples

Use XML `<example>` tags to separate examples from instructions. Claude parses these
unambiguously. Include 1-3 realistic examples that are:

- **Realistic** — mirror actual use cases
- **Diverse** — cover different scenarios
- **Project-agnostic** — use generic names, not project-specific paths
- **Descriptive** — show reasoning, not just output

### References

For content exceeding the body:
- Place in `references/*.md` files alongside SKILL.md
- Reference clearly: "See `references/spec-clarification.md` for the full process."
- For large reference files (>300 lines), include a table of contents

## Validation

Run `validate-skill.py` after generating any SKILL.md:

```bash
python3 ~/.copilot/tools/validate-skill.py path/to/SKILL.md
```

## Anti-patterns

| Pattern | Problem | Fix |
|---------|---------|-----|
| Vague description | No trigger words → never activates | Add specific trigger phrases |
| Flat 20-item checklist | Model loses track | Group into named phases |
| "MUST/ALWAYS/NEVER" without reason | Brittle compliance | Explain reasoning |
| Project-specific hardcoding | Breaks on other projects | Use `references/` |
| 800-line SKILL.md | Wastes context | Split: body ≤500 + references |
| No examples | Model guesses wrong | Add 1-3 `<example>` blocks |
| Redundant instructions | Noise | Remove; test if behavior changes |
